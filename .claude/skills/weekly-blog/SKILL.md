# 週報ブログ記事生成スキル

Notionの週報メモ（箇条書き・未整形テキスト）を、blog.milkmaccya.com で公開できるクオリティのMarkdown記事に変換するスキル。

**重要**: このブログはポートフォリオとしても機能し、社内外のエンジニアや採用担当が閲覧する可能性がある。文体・構成の品質は妥協しないこと。

## 会話フロー

### Step 0: Notion Inboxからの週報メモ生成（オプション）

ユーザーが「今週の週報を作って」と依頼し、まだ週報メモがNotionにまとまっていない場合、このステップを実行する。

**データソース:** Notion Inbox データベース（ID: `2b93611beab880fc8a4cfdb981797149`）

**Notion データ取得方法（優先順位付き）:**

以下の順で試行し、最初に成功した方法を使う:

1. **MCP（推奨）**: `notion-query-database-view` や `API-query-data-source` などのNotion MCP ツールでクエリを試みる。成功すればこれが最も簡潔。
2. **curl で Notion API 直接叩く**（フォールバック）: MCPが失敗した場合（プラン制限、MCP実装バグ等）に使用。APIトークンは `.mcp.json` の `OPENAPI_MCP_HEADERS` から `Authorization` ヘッダーの Bearer トークンを読み取る。

**注意:** MCPが失敗する場合、エラー内容を確認し、MCPのバージョンアップで解消されていないか検討すること。

**日付フィルタのタイムゾーン注意:**
Notionの `created_time` はUTC基準で保存されている。日本時間（JST = UTC+9）で対象週の月曜日が開始日の場合、UTC換算で前日になるエントリを取りこぼす。そのため **対象週の月曜日の1日前（日曜日）** を `on_or_after` に指定し、取得後にJST変換してフィルタする。

**curl フォールバックの取得コマンド:**

```bash
curl -s -X POST 'https://api.notion.com/v1/databases/2b93611beab880fc8a4cfdb981797149/query' \
  -H 'Authorization: Bearer <token from .mcp.json>' \
  -H 'Notion-Version: 2022-06-28' \
  -H 'Content-Type: application/json' \
  -d '{
    "filter": {
      "property": "日付",
      "created_time": {
        "on_or_after": "YYYY-MM-DD"  // 対象週の月曜日 - 1日（日曜日）
      }
    },
    "sorts": [{"property": "日付", "direction": "ascending"}],
    "page_size": 100
  }'
```

curlが失敗した場合（API仕様変更等）、Notion API の最新ドキュメントを確認し、エンドポイントやパラメータを修正すること。

取得後、各エントリの `created_time` をJST（+9時間）に変換し、対象週の月曜日以降のエントリのみを残す。

**手順:**
1. 上記の方法で対象週のInboxエントリを収集する（UTC→JSTフィルタ込み）
2. 取得したエントリを週報のセクションに分類する。タグがある場合は参考にするが、**タグがない・付け忘れのエントリも多い**ため、名前・要約・本文の内容から類推して分類すること:
   - 家族・生活: 育児、子供、妻、シッター、体調、生活の出来事など
   - 仕事・キャリア: アサイン、プロジェクト、復帰、チーム、組織など
   - 技術・インプット: 技術記事、ツール、ライブラリ、AI、Frontend、公開URLありなど
   - 振り返り・その他: 感情、気づき、自己啓発、買い物など
   - 除外候補: 「後で読む」タグのみ付いたもの（未消化のため）
3. 各エントリの「名前」「要約」「URL」を読み取り、セクション別に箇条書きで整理する
4. 整理した内容をユーザーに提示し、追加情報や修正を確認する

**注意:** Inboxのエントリだけでは育児エピソードの感情や文脈が不足することが多い。Step 0の出力はあくまで「骨格」であり、ユーザーとの会話でディテールを補完すること。特に以下を確認する:
- 「家族・生活」で書きたいエピソードの優先順位
- その週の体調・気分（振り返りのトーンに影響する）
- メモにない出来事（口頭で追加されることが多い）

### Step 1: 素材を受け取る

ユーザーからNotionの週報メモ（テキスト貼り付け or Notion MCP経由 or Step 0の出力）を受け取る。

### Step 2: 匿名化チェック

受け取ったテキストに以下の実名・個人情報が含まれる場合、**必ず**以下の表記に置換する。

| 入力に含まれうる表現 | 置換先 |
|---|---|
| つむちゃん / 紬 | 長女（2歳）※年齢は時期に応じて更新 |
| あやと / いおり（双子） | 双子の兄 / 双子の弟 / 双子 |
| さとこ / さっちゃん | 妻 / パートナー |
| 国永さん（シッター） | シッターさん |
| 具体的な病院名（日大板橋病院など） | 大学病院 / 専門外来 |
| Notionの内部リンク（notion.so/...） | 削除する |
| 家族の写真の直接的な説明 | 抽象化する |

**例外**: Qiita、Zenn、公式ドキュメント、GitHub等の公開URLはそのまま残す。

### Step 3: 下書きを生成する

テンプレートと文体ルールに従い、完全な下書きをチャット上に出力する（ファイルはまだ作らない）。
下書きを出した後、以下のように確認する:

「下書きを出しました。気になるニュアンスや、追加したいディテールがあれば教えてください。」

### Step 4: 対話的なニュアンス補正

ユーザーはカジュアルな口語でフィードバックすることが多い。そのまま文体に変換すること。

**よくあるフィードバックのパターンと対応:**

| ユーザーの指摘（例） | 対応方針 |
|---|---|
| 「比喩が大袈裟」「AIっぽい」 | 該当箇所を事実ベースの淡々とした表現に書き直す |
| 「〜は僕がやったんだよ」「〜は妻だよ」 | 主語や役割分担の事実を正確に反映する |
| 「ここもうちょっとこう」+ 口語での補足 | 補足内容を「だ・である」調に変換して組み込む |
| 「これも書いて」 | 新しい情報を適切なセクションに追加する |
| 「トーンが違う」「こんなこと言わない」 | 過去記事のリファレンスを参照してトーンを再調整する |

**対話のスタンス:**
- ユーザーの口語的なフィードバックを素早く反映する。一度のフィードバックで複数箇所に影響がある場合はまとめて修正する。
- 修正後は変更箇所を簡潔に伝え、次の確認を促す。冗長な説明は不要。
- 「OK」「いいね」「これで」など承認の言葉が出るまで繰り返す。

**NG表現の自動検出:**
下書き生成時に以下のパターンが含まれていないか自己チェックする。含まれていた場合は生成段階で除去する:
- 「胸が熱くなる」「心が震える」「感動に包まれる」→ 感情の過剰表現
- 「朝の戦場」「修羅場」（文脈なく使われる場合）→ 安易な比喩の乱用
- 「〜に違いない」「まさに〜と言えるだろう」→ 断定風の推論
- 「社会の一員としての成長を感じ」→ 抽象的で空疎な総括
- 「素晴らしい」「最高の」「かけがえのない」の多用 → ポジティブの押し売り

**OK表現の方向性（代替として）:**
- 感情は具体的な行動・状況で示す（「嬉しくなった」→「自分も嬉しくなった」ぐらいのシンプルさでOK）
- 比喩は筆者が実際に使っているもの（「デンプシーロール」「オペレーションの余白」等）に限定する
- 率直さ重視（「正直しんどい」「まだ慣れない」のような飾らない言葉）

### Step 5: Markdownファイルとして出力

OKが出たら `/mnt/user-data/outputs/` に `YYYY-MM-DD.md` として保存し、`present_files` で渡す。

## Frontmatter 仕様

```yaml
---
title: '[週報] YYYY/MM/DD週'
description: 'その週を象徴する一言（30〜50文字）'
pubDate: 'YYYY-MM-DD'        # 公開日（通常は週末の土曜 or 日曜）
heroImage: '../../../assets/YYYY-MM-DD.webp'
---
```

- description は検索結果やOGPに表示される。キャッチーかつ内容を正確に反映すること。
- heroImage のパスは相対パスで固定。ファイル名は週の開始日。

### 構成の補足

- **家族・生活** → **仕事・キャリア** → **技術・インプット** → **振り返り** の順序が基本。読者はこの構成に慣れている。
- **仕事・キャリア** は見出し2として常に含める。復帰後は特に重要なセクション。仕事がない週（育休中など）でも、キャリアに関する思考やキャッチアップの話題があれば記載する。
- **技術・インプット** は `<details>` タグで折りたたみにする。
- 必要に応じて「運動・健康」「その他メモ」「今週の買い物」セクションを追加してよい。
- **4コマ漫画** はGeminiのnano banana proで生成したもの。ない週もある。ある場合は「その他」セクション内に配置する。画像ファイルは `../../../assets/YYYY-MM-DD/YYYY-MM-DD-4koma.webp` の命名規則。画像ファイル自体の生成・選定はユーザーが行う。SKILL側では配置場所とマークアップのみ担当する。
- その他の figure 画像がある場合は、以下の形式で埋め込む:

```markdown
<figure>

![alt text](../../../assets/YYYY-MM-DD/YYYY-MM-DD-name.webp)

<figcaption class="text-center text-sm text-gray-500 dark:text-gray-400 -mt-4">キャプション</figcaption>
</figure>
```

## 文体・トーンのルール

### 基本文体

- **「だ・である」調**の断定的なブログ口語体。
- 一人称は「自分」を基本とする。文脈に応じて省略も可。
- 1文は短めに。冗長な修飾を避け、テンポよく読めるように。

### パートごとのトーン使い分け

**技術パート:**
- エンジニアらしく論理的かつ情熱的に。
- 「〜に直結すると感じた」「〜実戦投入の価値が極めて高い」のような断定と評価。
- Frontend Manager / SIer出身 / 大規模Next.js経験というバックグラウンドを反映した視点。
- 技術への好奇心と実務への接地感を両立させる。

**育児パート:**
- 大変さを隠さず、かつ子供への愛情が自然に伝わる表現。
- ユーモアは積極的に活かす（例：「デンプシーロール」「回避不能なパンチ」）。
- ただし過剰な比喩や大袈裟な表現は避ける。事実ベースの落ち着いたトーン。
- 育児をエンジニアリング用語で表現しない（「運用フロー」「リファクタリング」「デバッグ」等は使わない）。過去記事にはあるが、今後はやめる方針。

**仕事・キャリアパート:**
- 復帰後のリアルな所感を率直に。組織の変化への驚き、キャッチアップの焦り、プロジェクトへの手応えなど。
- 社内の具体的な機密情報（プロジェクト名、人名、事業戦略の詳細）は書かない。「重要プロジェクト」「組織が流動的」のように抽象化する。
- Frontend Manager経験を持つプレイヤーとしての視点。AIツールの実務活用、チーム開発、技術選定などは仕事パートで深掘りしてよい。
- 「育児と仕事の両立」のリアリティが自然に出るトーン。無理に美化しない。

**振り返りパート:**
- 最も「生身の声」が出るパート。
- 「正直しんどい」「まずは生き残ること」のような率直さ。
- 次週への意気込みは控えめに。「全部はやらない」「1テーマだけ前に進めば十分」のような自分への許可のトーンが特徴。

### 見出しの付け方

- 単なる「双子について」ではなく「双子の睡眠改善とデンプシーロールな日々」のように具体性を持たせる。
- 対比構造やユーモアを含めると良い（例：「寝なければ嘆き、寝すぎれば心配する」）。

## 絶対にやってはいけないこと

1. **内容を推論・創作しない。** メモにない事実を足さない。
2. **過度にポジティブな結びにしない。** 「素晴らしい1週間でした！」のようなAI臭い締めは厳禁。
3. **箇条書きの羅列にしない。** エピソード形式で書く。
4. **匿名化ルールを破らない。** 実名が1つでも残っていたら即修正。
5. **技術トピックを浅く流さない。** 各トピックに「なぜ自分にとって重要か」の一言を必ず添える。
6. **リンクにMarkdown記法を使わない。** ブログには `remark-link-card-plus` プラグインが入っており、裸のURLが自動でリンクカードに変換される。`[テキスト](URL)` 形式ではなく、URLをそのまま単独行に置くこと。補足が必要な場合は説明文の後にURLを改行で配置する。

## 過去記事のリファレンス（トーンの参考）

### 良い育児パートの例

> 双子がなかなか寝付かない横で、長女が赤ちゃんたちの様子をじっと気にしている。その光景には愛おしさを感じるが、現実問題として親の体力は限界に近い。デンプシーロールのごとく左右から押し寄せる育児の波をどう捌くか。もはや「気合」だけでは乗り切れない、戦略的な休息が必要だと感じている。

→ 愛情 + 体力の限界 + ユーモア + 冷静な分析が1パラグラフに共存している。

### 良い技術パートの例

> AOM（Accessibility Object Model）経由でDOMを扱わせる手法は目から鱗だった。AIエージェントにブラウザ操作を委ねる具体的なイメージがより鮮明になった。

→ 具体的な技術名 + 自分の学び + 展望が簡潔にまとまっている。

### 良い仕事・キャリアパートの例

> 画面越しに話した上長は以前と変わらない雰囲気で、その変わらなさに非常に安心した。知っている人が変わらずそこに居てくれるだけで、復帰へのハードルが少し下がった気がする。

→ 具体的な人物名を出さずに、復帰の心情を率直に描写。「変わらなさ」への安堵が自然に伝わる。

> AIを使った開発フローが当たり前になっていて（これは想定内）、「スクラムもどき（？）」のような新体制へ移行中らしい。

→ 機密に触れない範囲で組織の変化を記録。括弧書きのカジュアルな疑問符がこのブログの温度感。

### 良い振り返りの例

> 正直なところ、体調が万全ではない中での年末進行はしんどい。先週からの疲れが抜けきらず、「回復しきる前に次の波が来る」という感覚だ。読みたい資料が積み上がっていくことへの焦りもあるが、今は「全部やる」ことを諦める勇気が必要だろう。

→ 率直な弱音 + 自己分析 + 「諦める勇気」という自分への許可。これがこのブログの振り返りのトーン。
